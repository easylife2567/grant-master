# Palladium: A DPU-enabled Multi-Tenant Serverless Cloud over Zero-copy Multi-node RDMA Fabrics

## 1. 论文基本信息

- **作者**：Shixiong Qi (Univ. of Kentucky), Songyu Zhang (UC Riverside), K. K. Ramakrishnan (UC Riverside), Diman Z. Tootaghaj, Hardik Soni, Puneet Sharma (Hewlett Packard Labs)

- **来源 query**：Q1

## 2. 研究问题与动机

### 核心问题
Serverless 计算的数据平面因 kernel-based 网络开销而性能低下，现有 RDMA 方案（FUYAO、RMMap）虽提升了跨节点性能，但**缺乏多租户 RDMA 资源隔离**。同时，DPU（SmartNIC）offload 面临"弱核处理"和 CPU-DPU 跨处理器数据搬移瓶颈。

### 动机
- Serverless 函数链的松耦合导致重复数据平面处理，性能损失严重
- 共享内存方案仅在节点内有效，跨节点仍需 kernel 网络栈或非零拷贝 RDMA
- RDMA 多租户问题：用户函数不应直接访问 RNIC QP（安全风险），需要在 DPU 上运行受信任的 Network Engine 作为中间层
- DPU 的 on-path 模式导致额外数据搬移（DPU SoC DMA 速度慢），CPU-DPU 数据搬移可使性能下降 1.54x
- HTTP/TCP 到 RDMA 的协议转换若延迟到 worker node 执行，性能损失高达 11.4x

## 3. 核心方法

### 3.1 DNE：DPU Network Engine（反向代理）
- **Off-path 模式**：DNE 不触碰数据路径，只做控制平面的 QP 管理。数据由 RNIC DMA 直接写入 cross-processor 共享内存（主机 DRAM）
- **Cross-processor 共享内存**：利用 NVIDIA DOCA mmap 将主机统一内存池映射到 DPU ARM 核心，消除 on-path 数据搬移
- **Run-to-completion 事件循环**：消除 DPU 弱核上的上下文切换和调度开销

### 3.2 多租户 RDMA 隔离
- DNE 作为节点级 RDMA 反向代理，独占 QP 管理权限，用户函数不直接操作 QP
- **DWRR 调度器**（Deficit Weighted Round Robin）：按租户权重公平分配 RNIC 带宽
- **Shadow QP 机制**：维护活跃/非活跃连接池，只限制活跃 QP 数量避免 RNIC cache thrashing
- 按租户隔离统一内存池（per-tenant DPDK memory pool，通过 file-prefix 机制）

### 3.3 双向 RDMA + Token Passing 的无锁零拷贝
- 核心论证：**双向 RDMA 更适合 serverless 分布式数据平面**——因为它是 receiver-aware 的（接收方显式 post buffer），避免单向 RDMA write 的"发送方无视接收方"导致的数据竞争
- 性能验证：双向 RDMA 比 OWDL（单向+分布式锁）快 2.3x，比 OWRC（单向+接收方拷贝）快 1.3x
- 节点内：eBPF SK_MSG 传递 buffer descriptor
- 跨处理器：DOCA Comch-E（event-driven）传递 buffer descriptor

### 3.4 集群入口早期协议转换
- HTTP/TCP → RDMA 转换在集群边缘 ingress gateway 完成，worker node 只处理 RDMA 流量
- 基于 DPDK F-stack + NGINX 实现高性能 TCP/IP 处理
- 水平自动伸缩（CPU 利用率 >60% 扩容，<30% 缩容）

## 4. 与课题相关性

### 与本轮精读目标的关联
Palladium 的核心相关点在于其 **DNE 作为 RDMA 多租户隔离的软件中间层**——通过 DWRR 调度实现跨租户公平带宽分配。这为理解"RDMA 多租户性能隔离"提供了第三种技术路线（vs Harmonic 的硬件 FPGA + CNP 注入，vs PeRF 的纯软件抢占）。

### 对假设的验证
- H1（"多租户RDMA网络共享下存在不可忽视的相互干扰"）：Palladium 的实验证明了**无隔离时（FCFS DNE）租户间 starvation 现象显著**——先到的突发租户可完全占据带宽（印证干扰存在性）
- H2（"现有方案未充分考虑训练场景的通信模式特征"）：Palladium **完全针对 serverless 场景**，权重策略是静态的，不感知训练工作负载的迭代通信特征

### 对本课题的技术启示
- DNE 的 off-path + cross-processor 共享内存架构为训练场景提供了启发：训练任务的网络调度可部署在 DPU 上（不占用 GPU 服务器 CPU），通过 DWRR 权重动态调整实现训练任务的通信调度
- Token-passing + two-sided RDMA 的确定性 buffer 所有权模型可能天然适合训练任务的同步通信（每个迭代的通信阶段有明确的收发模式）
- 但其 DWRR 权重策略是静态的——改进方向：感知训练迭代阶段动态调整权重

## 5. 实验与评估

### 实验环境
- 4台服务器，每台 2×40核 CPU + 500GB RAM
- 2台 worker node 各配备 NVIDIA Bluefield-2 DPU（集成 ConnectX-6 RNIC）
- 200Gbps 交换机互联

### 主要结果
- **4.1.1 Off-path vs On-path**：Off-path（cross-processor 共享内存）比 on-path RPS 提升 30%，延迟降低 20%
- **4.1.2 RDMA 原语选择**：双向 RDMA 比 OWDL 快 2.3x，比 OWRC-Best 快 1.3x
- **4.1.3 早期协议转换**：Palladium ingress 比 kernel TCP ingress 延迟降低 3.4x，吞吐提升 3.2x
- **4.2 多租户隔离**：DWRR 调度精确按权重（6:1:2）分配 110K RPS 总带宽，FCFS 则导致 starvation
- **4.3 真实应用（Online Boutique）**：Palladium(DNE) 比 NightCore 吞吐高 20.9x、延迟降低 21x，比 FUYAO-F 高 2.1-4.1x，比 Palladium(CNE/CPU版) 高 1.3-1.8x
- **效率**：仅消耗 2 个 wimpy DPU 核心（相当于节省 7 个 CPU 核心）

## 6. 局限与 Gap

### 论文明确指出的局限
1. **仅支持固定大小内存池**：当前只实现固定大小 buffer，对于变长消息的 serverless payload 是限制
2. **Comch 通道瓶颈**：DOCA Comch 的实现使用 epoll（闭源），限制了 Comch-P 的可扩展性
3. **仅支持可靠的 RC 连接**：未评估 UD/UC 场景下的性能和多租户行为（虽然 §?? 提到了 Shadow QP 只适用于 RC）
4. **未解决跨安全域通信**：跨租户通信仍需显式 CPU 数据拷贝
5. **Ingress 伸缩短暂中断**：自动伸缩过程触发 worker 进程重启

### 论文未涉及但对本课题重要的 gap
1. **完全针对 serverless 场景**：工作负载模型（函数链、微服务）与分布式训练（大规模同步集合通信）差异巨大
2. **DWRR 是静态权重方案**：不感知工作负载动态变化，对训练任务的 burst/空闲交替模式无自适应
3. **隔离粒度在租户级（function chain）而非流/任务级**：训练场景可能需要更细粒度的通信调度
4. **未评估 GPU Direct RDMA**：训练场景中 GPU 与 RNIC 直连的 PCIe 拓扑与 DPU 的 off-path 架构可能存在冲突
5. **跨节点协调仅限 DNE 间连接池管理**：无全局通信调度视角——各节点 DNE 独立决策
6. **RDMA QP 映射是粗粒度的**：每个租户的所有 RCQPs 共享一个 RQ——训练中不同集合操作（AllReduce vs data loading）可能需要不同 QP 优先级
