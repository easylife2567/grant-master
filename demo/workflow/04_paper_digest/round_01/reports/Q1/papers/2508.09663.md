## 2. 研究问题与动机

### 核心问题
HPE Slingshot（200Gbps HPC 互联，用于多台顶级超算）的访问模型基于 Linux UID/GID 白名单认证，与 Kubernetes 容器化多租户部署**根本不兼容**——容器内 UID/GID 通过 user namespace 映射到 host，可被租户自由修改，导致 CXI 服务认证绕过。

### 动机
- Converged HPC-Cloud 是新兴范式：需同时满足云原生的租户隔离要求和 HPC 的最小化性能开销
- Slingshot 的 Virtual Network (VNI) 提供 L2 网络隔离域，CXI Service (SVC) 提供成员级 NIC 资源访问控制——但均为静态配置、基于 Linux UID
- Kubernetes 抽象了身份管理：所有容器以同一 UID 运行，无法通过现有机制区分
- 已有 InfiniBand+SR-IOV 方案不适用于 Slingshot（不支持 SR-IOV）

## 3. 核心方法

### 3.1 三层改造

**(A) CXI 驱动和库：netns-based 认证**
- 扩展 CXI kernel driver 增加第三种成员类型：**network namespace (netns) member**
- 认证时通过 procfs 提取请求进程的 netns inode（而非 UID/GID），因为 netns 由容器运行时管理、应用无法修改
- 同步扩展 libcxi（用户态库）和 libfabric（网络抽象层）

**(B) CXI CNI 插件**
- 实现为"链式"CNI插件（可与 Flannel/Cilium 等共存）
- ADD阶段：(1) 提取容器 netns inode → (2) 从 VNI Service 获取 VNI → (3) 创建 CXI Service 关联 netns inode 和 VNI
- DEL阶段：清理对应 CXI Service
- 仅对带 vni:true 注解的容器执行操作

**(C) VNI Service（Kubernetes 原生）**
- **VNI Controller**（基于 Metacontroller Decorator Controller）：监听带 vni 注解的 Kubernetes Job，自动创建 VNI CRD 实例
- **VNI Endpoint + SQLite 数据库**：全局管理 VNI 分配/释放，所有操作用 ACID 事务保证原子性
- 两种所有权模型：
  - **Per-Resource VNI**：每个 Job 自动分配一个独占 VNI，Job 终止后 30s 宽限期回收
  - **VNI Claim**：手动创建 VNI Claim 资源，多个 Job 可共享同一 VNI（跨 Job 通信场景）

### 3.2 安全属性
- netns 由容器运行时控制，应用无法伪造
- VNI 独占分配保证租户间网络完全隔离
- Slingshot Rosetta 交换机可配置严格 VNI 强制路由，NIC 间仅 VNI 匹配的可通信

## 4. 与课题相关性

### 与本轮精读目标的关联
本文是批次中**唯一关注 HPC 网络硬件到云原生软件栈桥接**的工作。与 Harmonic/Husky/PeRF 关注 RNIC 内部资源公平隔离不同，Slingshot 论文关注的是**通信域的租户间隔离**（谁可以跟谁通信），而非**资源使用的隔离**（占用多少带宽/PU/cache）。

### 对假设的验证
- H1（"多租户RDMA网络共享下存在不可忽视的相互干扰"）：Slingshot 通过 VNI 解决的是**安全隔离**（防止恶意租户监听/注入），而非**性能干扰**。论文未涉及共享 RNIC 资源时的相互性能影响
- H2（"现有方案未充分考虑训练场景的通信模式特征"）：完全不涉及。VNI 是静态的 L2 隔离域，不感知流量模式

### 技术启示
- VNI + netns-based 认证机制为训练任务提供了**容器级别的安全通信域隔离**——将训练任务的不同阶段（data loading vs gradient sync）隔离到不同 VNI 可防止跨任务干扰，但论文不涉及该场景
- CNI 插件的开销评估（延迟/吞吐 <1%，作业准入开销 <3.5%）表明 HPC 网络虚拟化可极低开销实现
- 与本批次其他论文的互补性：Slingshot 解决"谁能通信"，Harmonic/PeRF/OSMOSIS 解决"通信时如何公平共享资源"

## 5. 实验与评估

### 实验环境
- OpenCUBE 试点系统，2 个 Ampere Altra 节点
- k3s Kubernetes 部署
- 软件栈：OpenSUSE 15.5, libfabric 2.1.0 (patched), Open MPI 5.0.7, OSU Micro-Benchmarks 7.3

### 主要结果

**4.1 通信开销**
- osu_bw 吞吐量和 osu_latency 延迟：**开销 <1%，与 baseline（host 直连）无显著差异**
- vni:true（启用 netns+VNI）与 vni:false（全局 VNI 无隔离）之间也无差异
- 证明网络命名空间认证 + VNI 分配不引入可测量的通信性能开销

**4.2 作业准入开销**
- **Ramp Test**（逐渐增加到 10 jobs/s）：job admission 中位数延迟 overhead 3.5%，主要延迟来自 Kubernetes 控制平面本身
- **Spike Test**（500 jobs 一次性提交）：overhead 1.6%
- VNI 分配/释放 + CXI Service 创建/删除均不显著影响作业启动时间

## 6. 局限与 Gap

### 论文明确指出的局限
1. **仅实现通信域隔离，不提供性能隔离**：VNI 保证租户间不可见/不可通信，但共享 NIC 时的带宽争用仍存在
2. **不支持 VNI 间 QoS**：虽然 Slingshot 硬件支持 Traffic Class（流量类别），但论文未实现按 VNI 的 QoS 配置
3. **netns 仅适用于 Linux 容器**：不支持 VM 场景
4. **无跨集群 VNI 协调**：单个 Kubernetes 集群内的方案
5. **仅评估 2 节点、千/百包量级的 MPI 微基准和轻量容器**：实际 HPC 规模未验证

### 论文未涉及但对本课题重要的 gap
1. **完全不涉及 RNIC 内部资源争用的性能隔离**：通信域隔离是安全维度的（CIA），性能隔离是 QoS 维度的——本论文只解决前者
2. **不感知分布式训练通信模式**：VNI 是静态配置的 L2 网络隔离，与工作负载特征无关
3. **VNI 资源分配策略是 first-come-first-served**：不基于租户优先级或工作负载需求动态调整
4. **无与 RNIC 微架构隔离方案的集成讨论**：如 Harmonic 的 PIPS + CNP 方案如何与 Slingshot VNI 结合
