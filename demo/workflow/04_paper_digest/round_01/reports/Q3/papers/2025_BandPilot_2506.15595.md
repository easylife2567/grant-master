# BandPilot: Towards Performance- and Contention-Aware GPU Dispatching in AI Clusters

## 2. 研究问题与动机

在多租户 AI 集群中，GPU 调度器选择哪些物理 GPU 分配给租户，直接决定了该租户的集体通信带宽上限。现有调度器依赖**拓扑紧凑度（topology-compactness）启发式**（如 Slurm 的 topology-aware 策略），假定 GPU 越紧凑通信性能越好。论文揭示这一假设在真实 H100 集群中经常失效：

- **反直觉案例**：8 GPU 请求时，平衡的 4+4 跨节点分配（337 GB/s）比紧凑的 6+2 分配（153 GB/s）带宽高 2.2 倍
- **根因**：紧凑分配导致单个 NIC 被 8 个 GPU 的流量所饱和，成为瓶颈

**核心问题**：如何设计一个性能感知且争用感知的 GPU 调度器，在组合爆炸的分配空间中实时找到近似最优的 GPU 子集？

---

## 3. 核心方法与技术方案

BandPilot 采用**学习+搜索**的闭环架构。

### 3.1 层次化带宽替代模型（Hierarchical Transformer）

- **Stage-1 节点内查表**：对 8-GPU 节点内所有 255 种组合进行穷举 NCCL 测量，存入键值字典
- **Stage-2 节点间 Transformer 预测**：输入为每个节点上分配的 GPU 数量及该节点内的查表带宽，通过 6 层 Transformer encoder + 3 层 MLP 预测跨节点集体通信带宽
- **数据效率**：仅需 250 个训练样本即可达到 R²>0.95，MAPE<5%

### 3.2 争用感知预测器

- 将活跃作业的 GPU 分配和带宽需求抽象为流量画像 τ
- 构造"超级分配"（super allocation）：候选分配与共享节点的活跃作业合并，用 ˆB(S) 模型估算共享瓶颈容量
- 采用比例共享模型：若总需求超过瓶颈容量，按需求比例分摊带宽

### 3.3 混合搜索算法

- **均衡驱动启发式（EHA）**：基于"均衡分配优于紧凑分配"的经验法则，快速生成备选分配
- **剪枝树搜索（PTS）**：自上而下逐个淘汰"拖后腿"的 GPU（O(|A|²-k²) 复杂度）
- 两个算法取最优结果输出

### 3.4 系统架构

- 离线：NCCL 测量 → 节点内字典 + 节点间训练数据 → 模型训练
- 在线：请求到达 → 控制接口获取可用 GPU 集 A 和流量画像 τ → 混合搜索 → 输出近似最优分配

## 4. 与课题相关性分析

**课题核心关注**：RDMA 多租户性能隔离和分布式训练通信调度。

**高度相关**：
- BandPilot 直接回应了课题核心问题："如何在多租户场景下为每个任务选择最优 GPU 子集以最大化通信性能？"
- 论文从**黑盒预测**的角度处理多租户争用，不需要了解租户工作负载细节，这与课题中 NCCL 不感知上层训练任务的假设一致（H1 假设）。
- 层次化带宽模型在 32-GPU H100 集群上达到 92-97% 的带宽效率，证明了数据驱动方法的可行性。
- 与 vClos 互补：vClos 通过空间隔离消除争用，BandPilot 在争用不可避免的情况下智能规避；与 Arnold 互补：Arnold 优化单个大规模作业的内部拓扑对齐，BandPilot 优化多作业间的 GPU 粒度的放置。

**潜在 gap**：
- 实验规模局限（32 GPU），是否可扩展到数百 GPU 的大规模集群尚待验证。
- 争用模型假设比例共享，但真实 RDMA 网络的 QoS 行为（DCQCN 拥塞控制、PFC 反压）可能显著偏离线性比例模型。
- 仅考虑 all-gather 操作（16MB payload），是否适用于其他集体通信操作（all-reduce, all-to-all）未充分验证。

## 5. 关键实验与结论

1. **模型精度**：250 样本训练，H100 集群 R²>0.95，Het-4Mix 集群 MAPE<5%

2. **调度性能**（H100 集群）：
   - 无争用（τidle）：GBE 96.4%
   - 中等争用（τmod）：GBE 92.9%
   - 重争用（τhvy）：GBE 93.7%
   - 相比 Topo 基线提升约 20%（H100）和 40%（Het-4Mix）

3. **系统开销**：
   - 搜索延迟 <2.5s（32-GPU 集群）
   - 模型存储约 354 KB + 节点内字典 12 KB/节点
   - 离线测量时间：10-20 分钟/节点（可并行）

4. **消融实验**：
   - 层次化设计比朴素 Transformer 数据效率高 3.2x（250 样本时 MAPE 5.67% vs 17.89%）
   - EHA 在同构集群表现好，PTS 在异构集群更关键，两者互补

---

## 6. 局限与 Gap

1. **小规模验证**：物理实验仅在 32-GPU H100 集群上，模型在数百至数千 GPU 规模下的泛化能力未经验证。Transformer 对输入规模的扩展性（O(n²) 注意力）可能成为瓶颈。

2. **争用模型过于简化**：比例带宽共享模型无法精确刻画 RDMA 网络中 DCQCN 拥塞控制、PFC 反压、Incast 崩溃等非线性行为。

3. **仅覆盖集体通信带宽**：未考虑延迟敏感型通信（如小消息 P2P）或非 NCCL 通信模式（如 DeepEP、Mooncake 等）。

4. **需要预测量**：每个新集群需要 10-20 分钟/节点的离线测量 + 模型训练，增加了部署成本。对硬件变化（GPU 更换、NIC 升级）需要重新测量。

5. **未考虑作业调度层面**：BandPilot 定位为 GPU 调度"原语"（primitive），不处理作业排队、优先级、饥饿等调度策略问题。

---

## 7. 对后续研究的启示

- **数据驱动的 GPU 调度是可行的**：BandPilot 证明即使从极少的 NCCL 测量数据中，也能学习到有用的通信性能代理模型，这对本批次中其他依赖预表征的方案（Arnold）是一个方法论上的印证。
- **均衡 > 紧凑**：关键洞察——跨节点负载均衡比拓扑紧凑度更重要——是传统的拓扑感知调度（Slurm topology-aware）的根本性缺陷。
- **层次化分解是通用设计模式**：BandPilot 的节点内/节点间分解与 Arnold 的 TP 优先规则（节点内 NVLink）、VCCL 的 intra/inter-node 分离设计一致，说明"利用节点内高带宽 + 优化节点间通信"是这一领域的共识设计策略。
