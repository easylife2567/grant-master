# vClos: Isolated Scheduling for Distributed Training Tasks in GPU Clusters

## 2. 研究问题与动机

在 Leaf-Spine 网络架构的多租户 GPU 集群中，ECMP 哈希冲突导致不同 DML 任务的流竞争同一网络端口，造成通信时间和等待时间显著增加。论文通过 32-GPU V100 集群的实验表明：

- 即使选择最优哈希因子组合，仍有平均 31.5% 的概率发生网络争用
- 大集群（2048 GPU）下 6 流竞争概率为 3.3%
- 极端情况下，两流竞争可导致训练吞吐量下降约 60%（DLRM/MoE 类模型）

**核心问题**：如何在多租户 Leaf-Spine GPU 集群中消除网络争用，实现不同分布式训练任务之间的通信隔离？

---

## 3. 核心方法与技术方案

vClos 的核心思路是**为每个 DML 任务分配一个独占的虚拟 Clos 子拓扑**，从根本上消除任务间网络争用。

### 3.1 Leaf-wise Permutation Traffic Pattern（理论基础）

论文形式化证明了：大多数 DML 通信算法（Ring all-reduce、HD、Pipeline P2P、AlltoAll Pairwise Exchange）遵循"Leaf-wise 排列流量模式"。在此模式下，Source Routing 策略可以保证零争用：

- **Source Routing**：每个 Leaf 交换机将 server-facing 端口与 spine-facing 端口建立一对一映射；源 GPU 决定数据包经过的 Spine 交换机
- **争用自由条件**：源 Leaf 不同的两个流不会经过相同的 Spine 交换机到达各自目的 Leaf

### 3.2 vClos 调度算法（三阶段）

- **Stage 0**（单机内调度）：若 N_i <= T（单机 GPU 数），优先排入同一服务器
- **Stage 1**（单 Leaf 内调度）：优先将任务排在同一 Leaf 交换机下，减少 Spine 端口占用
- **Stage 2**（ILP 求解虚拟 Clos）：通过整数线性规划（ILP）在 Leaf 和 Spine 间构建独占虚拟子拓扑，平均求解耗时约 1 秒（2048 GPU 集群）

### 3.3 OCS-vClos（光学电路交换增强）

引入 MEMS-OCS 层来动态重配置 Leaf 和 Spine 之间的链路连接，减少网络资源碎片化：

- **Stage 2**：尝试通过 OCS 重配置，在单 Spine 内生成虚拟 Clos
- **Stage 3**：使用多 Spine 通过 OCS 链路重新布线

### 3.4 实现细节

- 基于 8 台华为 G5500 服务器 + 4 台华为 CE8850 交换机的 32-V100 测试床
- 通过 VRF 将物理交换机虚拟化为多个逻辑交换机
- ACL 下发实现 Source Routing；OCS 通过 Telnet 协议控制

## 4. 与课题相关性分析

**课题核心关注**：RDMA 多租户性能隔离和分布式训练通信调度。

**高度相关**：
- vClos 是**本批次中唯一系统性地从理论上证明并工程实现多任务通信完全隔离**的工作。
- 论文直接回应了 H2 假设（"多任务训练的网络干扰是真实可量化的"）：通过 32-GPU 实测数据证明 ECMP 哈希冲突导致明显的性能退化（VGG16/BERT 吞吐量下降，极端情况 AlltoAll 任务下降 60%）。
- 关键的"Leaf-wise Permutation Traffic Pattern"理论框架可推广到 RDMA/RoCEv2 环境下的通信调度。
- OCS 增强方案为未来可重构网络（如 OCS/Rotornet）在多租户 AI 集群中的应用提供了设计参考。

**潜在 gap**：
- 实验基于 100Gbps RoCE + V100 GPU，未验证更高带宽（400Gbps/800Gbps）和更强 GPU（H100/B200）下的效果。
- OCS 方案依赖特殊硬件（MEMS-OCS），其可靠性存疑（论文自述 OCS 原型对噪声敏感，需隔音棉和定期调校）。
- vClos 要求 N 为质数（prime number）才能完美构建虚拟 Clos，极端情况下需使用大于 N 的最小合数 Nnew，引入资源浪费。

## 5. 关键实验与结论

1. **测试床实验（32 GPU）**：
   - vClos 相比 ECMP 降低 Avg.JRT 约 21.4%，降低 Avg.JWT 约 37.3%
   - 大任务（16 GPU）受网络争用影响更大（JRT 增加 17.9%）
   - 增加 50% 冗余端口（rECMP）仍无法完全避免争用

2. **大规模仿真（512 GPU，5000 任务）**：
   - OCS-vClos 相比 Source Routing 降低 Avg.JWT 达 65.65%
   - OCS-vClos 的 Avg.JCT 仅比理论最优（Best）高 3.38%
   - ECMP 导致显著的 JWT 长尾分布，影响用户体验

3. **大规模仿真（2048 GPU）**：
   - vClos 的 Avg.JWT 仅比 OCS-vClos 高 12.06%，说明大规模下网络碎片化影响减弱

4. **关键结论**：
   - 消除网络争用可显著提升分布式训练性能，大任务获益更明显
   - OCS 可有效减少网络资源碎片化，但大规模下 vClos 已足够

## 6. 局限与 Gap

1. **N 为质数的约束**：虚拟 Clos 的构建要求任务所需 GPU 数为质数（> T 时），否则需要向上取整到最近的合数，引入资源浪费。论文通过 Helios 数据集分析声称大部分 N > T 的情形满足要求，但这在实际中并非总能保证。

2. **OCS 硬件可靠性**：论文使用的 MEMS-OCS 为第一代原型，对噪声敏感，频繁导致 RDMA 连接断开。测试床中需要使用隔音棉和定期透镜校准来维持稳定。

3. **通信算法实现效率低**：分层通信算法（HD、Hierarchical Ring）的实测效率较差——原因在于 NCCL 默认 4MB block size 在分层算法中导致 suboptimal 数据传输。

4. **仅考虑 Leaf-Spine 拓扑**：vClos 的设计依赖 Leaf-Spine 的可重排无阻塞特性（rearrangeably nonblocking），对于其他拓扑（如 rail-optimized、DGX SuperPOD 的 DragonFly）不直接适用。

5. **FIFO 调度器的局限性**：论文主要以 FIFO 调度为主，虽然在 SF 和 EDF 下也做了敏感度分析，但未探讨与更先进的作业调度器（如 Gandiva、Pollux）的协同。

6. **缺乏生产级部署验证**：全部大规模实验均为仿真（RapidNetSim），测试床仅 32 GPU，与工业生产级集群（数千 GPU）差距显著。

## 7. 对后续研究的启示

- **空间隔离是最彻底的解法**：vClos 的"独占子拓扑"策略与 BandPilot 的"智能选 GPU 子集"、Arnold 的"单任务拓扑对齐"构成了从强到弱的三种隔离方案。在实际部署中可根据任务优先级和规模灵活组合。
- **Source Routing 是低成本就近隔离手段**：即使不使用 vClos 的完全虚拟化，仅引入 Source Routing 即可将争用链路数从 L*S 降至 L（降低 S 倍）。
- **OCS 的引入值得关注**：随着 Google TPUv4/Jupiter 等系统中 OCS 的规模化部署，OCS-vClos 的拓扑重构思路可能在未来的 GPU 集群中获得硬件支持。
- **AlltoAll 是网络争用的最大受害者**：MoE 模型的 AlltoAll 通信对网络争用极度敏感（极端情况 60% 吞吐下降），这是本批次中所有论文都未充分解决的难点。
