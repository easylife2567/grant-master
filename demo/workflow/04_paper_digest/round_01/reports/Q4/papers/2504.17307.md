# An Extensible Software Transport Layer for GPU Networking (UCCL)

**作者**: Yang Zhou, Zhongjie Chen, Ziming Mao, ChonLam Lao, Shuo Yang, Pravein Govindan Kannan, Jiaqi Gao, Yilong Zhao, Yongji Wu, Kaichao You, Fengyuan Ren, Zhiying Xu, Costin Raiciu, Ion Stoica
**年份**: 2025 | **会议/期刊**: arXiv (preprint)
**引用数**: 10 | **关联度**: core | **优先级**: high

---

## 1. 论文概况

UCCL（Ultra-CCL）是一个可扩展的软件传输层，通过将 RDMA NIC 的控制路径与数据路径解耦，将传输控制逻辑（拥塞控制、负载均衡、丢包恢复）移至主机 CPU 软件执行。这使得 ML 工作负载可以获得硬件难以实现的传输创新——如多路径传输缓解流碰撞、接收端驱动的拥塞控制处理 incast、选择性重传代替 Go-Back-N。UCCL 以 NCCL 网络插件形式实现（28.4K 行 C++），可在现有 RDMA NIC 上运行，无需硬件修改。

---

## 2. 研究问题与动机

**核心问题**: 主机网络传输（基于 RDMA NIC）难以演进以适应快速变化的 ML 工作负载需求。硬件修改耗时长，造成应用需求与硬件优化之间的错配。

**六个实践案例驱动**:

1. **Meta 禁用 DCQCN**: DCQCN 在 LLM 训练（低流熵、高突发性）下效果不佳，Meta 在 NIC 中禁用 CC 转而实现应用层流量调度。DeepSeek 在 MoE serving 中也禁用 CC。
2. **Alibaba 流碰撞**: RDMA NIC 仅支持单连接单路径/流，导致流碰撞严重降低集合通信性能。Alibaba 被迫重新设计网络拓扑（rail-optimized dual-plane）。
3. **DeepSeek MoE incast**: Expert Parallelism 中某些 expert 接收 10x 于平均的负载，产生 incast。可通过接收端驱动 CC（如 EQDS）解决，但商用 NIC 不支持。
4. **应用-传输 codesign 不可行**: MLT（NSDI 2024）提出基于梯度重要性的半可靠传输，但无法集成到现有 RDMA NIC 硬件中。
5. **低效丢包恢复**: 老一代 NIC 的 Go-Back-N 在丢包下性能极差，但无法升级为选择性重传。
6. **异构 NIC 互操作**: 数据中心混合多代/多家 NIC 导致跨代通信带宽下降 2-33x（Alibaba 报告）。

## 3. 核心方法与设计

### 3.1 控制路径与数据路径分离

UCCL 的核心架构创新是将 RDMA NIC 的数据路径（GPU 数据传输）和控制路径（CC、LB、可靠性）分离：

- **首选 UC QP**: 绕过硬件 CC 和丢包恢复，利用 write_with_immediate 将 32-bit imm_data 用作传输控制头（含连接 ID、消息 ID、块序号等）
- **RC with CC disabled**: Broadcom NIC 等不支持 UC 时的备选
- **UD 作为最后手段**（AWS EFA）: 利用 scatter-gather 将控制头和数据载荷合并在一个 UD 包中传输，接收端自动分离到 CPU/GPU 内存。UD 无硬件分段/重组卸载，CPU 需处理更多工作
- **非 RDMA NIC**: 通过 AF_XDP（用户态快速包 I/O）在 UDP 上构建可靠传输

### 3.2 多路径传输

- 使用 256 UC/RC QP 或 16x16 UD QP 组合实现多路径（最多 256 条不同网络路径）
- 所有连接共享同一组 QP 以避免 QP 耗尽
- 软件维护 per-path RTT 状态，Power-of-Two 采样选路
- 意外发现：ML 工作负载的大消息传输有效摊薄了 QP 交换开销——60K QP 仅比 60 QP 降低 ~17% 带宽（RC），UC 几乎无影响

### 3.3 高效软件传输技术

- **连接拆分（Connection Splitting）**: 一条连接分给多个 CPU 核心，子连接各自维护 CC/LB 状态，支持 400Gbps 双向线速
- **控制合并（Control Coalescing）**: 以 32KB chunk（而非每包）为粒度做传输决策，降 CPU 开销
- **链式提交（Chained Posting）**: 一次 MMIO 写提交最多 32 个 WQE，减少 PCIe 开销（对 UD 特别重要）
- **乱序包处理**: 重排缓冲区放在 GPU 内存（非 NIC SRAM），利用 GPU 高带宽；reduction kernel 融合 scattered memcpy

### 3.4 三个可扩展性案例

**(1) 多路径 + 包喷洒**: 实现 CUBIC 和 Swift CC，per-path 或 global CC 状态均支持。UEC 标准的多路径协议可在 UCCL 上快速实现。

**(2) 接收端驱动 CC（EQDS）**: 专用 pacer 线程按固定速率分配 credits 给发送端。解决 MoE serving 中的 incast 问题。

**(3) 选择性重传**: 维持 GPU 内存中的重排缓冲区，使用 std::map 追踪任意数量的乱序包。比硬件 Go-Back-N 在丢包下恢复效率高得多。

## 4. 与课题相关性

结合本轮的 round_goal（"系统调研 RDMA 多租户性能隔离和分布式训练通信调度的现有方法"）：

- **高度相关**: UCCL 是本批次中唯一直接涉及"传输可编程性/可扩展性"的工作，直接回应对假设 H2（"软件可编程传输层是 RDMA 与 ML 训练深度融合的关键使能技术"）的验证。
- **多租户相关**: UCCL 未直接讨论多租户性能隔离，但其架构（软件控制平面 + 多路径 + 可编程 CC/LB）为多租户调度提供了天然基础。可编程 CC 策略可针对不同租户差异化配置。
- **通信调度**: UCCL 的多路径 LB + 接收端 CC 本质上是通信调度机制——通过 per-path RTT 的 Power-of-Two 选择实现路径级调度，通过 EQDS 的 credit 分配实现发送端速率调度。
- **与 OptiNIC/Celeris 对比**: OptiNIC/Celeris 从硬件侧简化 NIC（去除可靠性逻辑），UCCL 从软件侧接管控制平面。两者互补——UCCL 可在不改硬件的条件下实现传输创新，OptiNIC 提供了更彻底的硬件简化路径。两者结合（UCCL 软件控制 + 简化 NIC 数据路径）可能是最优方案。

## 5. 关键结果与评估

### 集合通信性能

**CX_IB（无拥塞，InfiniBand）**:
- UCCL UC/RC 与 ConnectX-7 硬件传输性能几乎相同（差距 <4%），验证软件传输可达到 ASIC 级性能

**CX_ETH（有流碰撞，Ethernet fat-tree）**:
- UCCL 比 ConnectX-7 提升 allreduce 最多 2.32x，all-to-all 最多 **4.54x**
- ConnectX-7 因单路径 + DCQCN 在大消息下严重性能下降；UCCL 多路径 + 智能 LB/CC 稳定扩展

**AMD（Broadcom Thor-2，rail-optimized topology）**:
- UCCL 仍优于 Thor-2：allreduce 最多 1.34x，all-to-all 最多 **1.93x**

**EFA（AWS p4d.24xlarge）**:
- UCCL 比 AWS SRD 提升 allreduce 最多 1.27x，all-to-all 最多 **3.27x**
- 原因：强大 CPU 核心 > SmartNIC 上 wimpy ARM 核心（特别是连接密集型 all-to-all）

### 应用性能

- ResNet 训练（EFA）：epoch 时间减少 1.07-1.11x
- DeepSeek-V3 MoE serving 模拟（EFA）：prefill 延迟减少 1.13x，decoding 延迟减少 **1.42x**

### 可扩展性案例

- **Incast 处理**: UCCL EQDS 比 InfiniBand CC 降低 incast 流量 P99 延迟 1.73x，permutation 流量 4.5-4.88x
- **丢包处理**: 1/16384 丢包率下仅 1% 性能损失（vs 文献中 RDMA 硬件传输 26-42% 损失）；1/256 丢包率下 30% 损失（vs 文献中 59-76%）

### 软件效率

- 1 CPU 核心饱和 400G 单向带宽（UC/RC），2 核心饱和 400G 双向
- CC 决策延迟 P99 在重负载下 <11us，ACK turnaround P99 <36us（与典型数据中心 RTT 10-40us 相当）

## 6. 局限与 Gap

1. **RDMA NIC 硬件依赖**: UCCL 在 UC/RC 模式下仍需 NIC 硬件分段/重组卸载才能达到线速。UD 模式消耗更多 CPU，对小消息不友好。

2. **QP 交换开销**: 虽被 ML 大消息摊薄，但当多条连接共享同一 NIC（如 8 GPU 共享 1 NIC）时 QP 数量需求倍增。256 QP × 多条连接可能触及 NIC 上限。

3. **多租户/多任务调度未系统评估**: UCCL 的可编程 CC/LB 为多租户调度提供基础，但论文未系统评估该场景。特别是多个训练任务共享网络时的 fairness、优先级划分、最小带宽保障等 QoS 机制未被讨论。

4. **CPU 额外开销**: 每 NIC 需 2 个额外 CPU 核心（接收端 CC 再 +1 核）。论文引用"128 核中仅 14.5% 利用率"佐证开销可接受，但在高 CPU 利用率场景下可能成为瓶颈。

5. **拥塞信号受限**: 现有 RDMA NIC 不向软件暴露 ECN marks 和 packet trimming 状态。UCCL 仅依赖 RTT 和丢包检测拥塞。论文讨论了未来的 HW-SW 接口改进方向。

6. **无商业化 NIC 集成验证**: UCCL 以 NCCL 插件形式运行，验证环境主要是 ConnectX-7、Thor-2、EFA。对其他商用 NIC（如 Intel IPU、Google Falcon、AMD Pensando）的支持情况未知。

7. **GPU-driven 通信兼容性**: 论文仅讨论 IBGDA CPU-assisted mode 可能性（牺牲 10% 性能），未实现或验证与 DeepEP 等 GPU-driven 方案的集成。

8. **安全性/隔离性**: 软件传输层运行在用户态进程中，在恶意或 buggy 应用场景下可能存在安全问题。多租户下的控制面隔离未被讨论。

## 7. 精读笔记

- **作者团队**: UC Berkeley（Ion Stoica 组）、清华（Fengyuan Ren 组）、Harvard、IBM Research、AWS、Broadcom。工业界作者参与暗示实用化潜力。
- **开源**: https://github.com/uccl-project/uccl，以 NCCL 网络插件形式发行。
- **与 OptiNIC/Celeris 的对比**: UCCL 和 OptiNIC/Celeris 从不同角度解决同一问题。UCCL 的哲学是"保持 NIC 硬件不变，通过软件实现传输创新"；OptiNIC/Celeris 的哲学是"改变 NIC 硬件以匹配 ML 需求"。UCCL 的优势在于即时可部署性，OptiNIC 的优势在于更彻底的硬件简化和更低的延迟。
- **UCCL 在 OptiNIC 论文中被引用**: OptiNIC 论文 Table 1 将 UCCL 列为比较对象，认可其"将整个传输控制面移至软件"的设计，但指出 UCCL 仍"保留选择性重传"而非取消可靠性。
- **多路径的关键价值**: 论文第 6.1 节结果展示了在 fat-tree 拓扑下 UCCL 多路径的巨大优势（all-to-all 最多 4.54x），说明单路径是当前 RDMA 在 ML 场景中的主要瓶颈之一。
- **关键参考文献**: EQDS (NSDI 2022)、Swift (SIGCOMM 2020)、SRNIC (NSDI 2023)、Flor (OSDI 2023)、MLT (NSDI 2024)、ZeroNIC (OSDI 2024)、SCR (NSDI 2025)
