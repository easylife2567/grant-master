## 2. synthesis 后的总体判断

经过一轮系统调研（25篇论文，5个方向），synthesis 得出了清晰的三层诊断：

1. **Layer 1（通信域安全隔离）和 Layer 2（性能资源隔离）已有成熟方案**——Slingshot K8s、Harmonic、PeRF、Palladium 等从不同维度解决了"谁可以通信"和"分配多少资源"的问题。
2. **Layer 3（工作负载感知的动态资源调度）是明确的空白**——所有6篇RDMA隔离论文将流量视为稳态流，所有7篇训练通信调度论文聚焦单作业优化。没有任何工作将"训练通信模式"作为"RDMA网络资源调度"的核心输入变量。
3. **三个高置信度Gap提供了清晰的切入点**——Gap 1（隔离不感知训练模式）、Gap 2（调度不考虑多租户）、Gap 3（CC/LB无训练专有优化），均获≥6篇独立论文支撑。

**核心判断**：本课题应聚焦于Gap 1+Gap 2的交叉地带——"训练通信语义驱动的RDMA多租户网络资源调度"，走纯软件路线（PeRF抢占原语为基础执行层），以青年项目的体量（3项研究内容，8-64节点实验验证）交付可验证的系统原型。

## 5. 核心问题筛选

**最终选择的核心问题**：在智算中心多租户共享RDMA网络场景下，如何利用分布式训练的通信语义（迭代周期、通信操作类型、ML容忍度）设计资源调度方法，实现网络性能隔离与高效利用？

**选择理由**：
- 直接对应 Gap 1 + Gap 2 的交叉空白——这是整个 synthesis 中最"不证自明"的创新空间
- 这两个 Gap 的证据链是整份 current_view 中最强的（共13篇独立论文交叉验证）
- "训练通信语义→RDMA调度"的范式尚未被任何工作提出——PeRF、Harmonic、BandPilot 各自的路线都没覆盖这个方向
- 青年项目的体量恰好适合解决这一问题（3项研究内容、8-64节点实验）
- 产出可明确对应（算法→调度策略论文、系统原型→开源代码、实验验证→benchmark数据）

**与topic对应**："面向高性能分布式训练的网络资源智能调度方法"——"智能"体现在利用训练通信语义做调度决策（而非盲目的速率限制），"网络资源调度"定位在RDMA的多租户资源分配。

## 7. 最终项目定位

**本项目主打**：以"训练通信语义感知"为核心理念，在不修改RDMA NIC硬件的约束下，设计纯软件的RDMA网络资源智能调度方法，实现多租户训练场景下的性能隔离与资源高效利用。

**本项目不主打**：RDMA硬件协议重新设计、全新训练通信调度算法、与集群调度器的深度集成改造。

**申请书主线表达**：面向智算中心多租户共享RDMA网络的分布式训练场景，针对现有RDMA性能隔离方案不感知训练通信模式导致隔离效率低、资源利用率差的问题，本项目提出训练通信语义驱动的RDMA网络资源智能调度方法——通过（1）在线感知训练任务的通信模式特征，（2）将通信语义编码为调度优先级和资源配额，（3）基于RDMA抢占原语执行差异化调度——实现多任务训练的网络性能隔离与资源高效利用。

## 8. 整体技术方案

### 总体思路

采用三层架构：**感知层**（从NCCL通信trace和RDMA计数器实时提取训练任务的通信模式）→ **决策层**（基于通信模式和历史性能数据的调度策略，分配QoS等级、QP资源和路径）→ **执行层**（利用PeRF抢占原语、QP速率控制和源路由实现调度策略）。

### 系统架构

```
┌─────────────────────────────────────────────────┐
│              调度决策引擎 (Decision)              │
│  ┌─────────────┐  ┌──────────┐  ┌────────────┐  │
│  │ 通信模式分类器│→│ 资源分配器│→│ 策略执行器  │  │
│  │(burst/空闲检测│  │(带宽/路径│  │(抢占命令生成│  │
│  │ 操作类型识别) │  │ /QP配额) │  │ /速率更新)  │  │
│  └─────────────┘  └──────────┘  └────────────┘  │
├─────────────────────────────────────────────────┤
│              通信感知层 (Perception)              │
│  ┌──────────────┐  ┌───────────────────────────┐ │
│  │ NCCL Trace   │  │ RDMA 计数器采集            │ │
│  │ 拦截器       │  │ (QP统计/拥塞信号/速率)     │ │
│  └──────────────┘  └───────────────────────────┘ │
├─────────────────────────────────────────────────┤
│              调度执行层 (Execution)               │
│  ┌──────────────┐  ┌──────────┐  ┌────────────┐ │
│  │ QP抢占管理器  │  │ 速率控制器│  │ 路径管理器  │ │
│  │(PeRF原语封装)│  │(令牌桶)  │  │(源路由)    │ │
│  └──────────────┘  └──────────┘  └────────────┘ │
└─────────────────────────────────────────────────┘
```

### 关键模块

| 模块 | 角色 | 方法 | 输入 | 输出 |
|------|------|------|------|------|
| M1: NCCL Trace拦截器 | 感知训练通信模式 | LD_PRELOAD劫持NCCL API调用，获取操作类型/消息大小/QP绑定 | NCCL集体通信调用 | 通信事件流(timestamp, op, size, QP) |
| M2: RDMA计数器采集器 | 感知网络状态 | 周期性读取RDMA QP计数器（发送字节/包数/重传/ECN标记） | /sys/class/infiniband/*/ports/*/counters/ | 网络状态快照 |
| M3: 通信模式分类器 | 识别任务通信阶段 | 滑动窗口+简单规则/轻量ML模型，识别burst/空闲/操作类型 | 通信事件流+网络状态 | 任务通信模式标签(burst_type, intensity, period) |
| M4: 资源分配器 | 计算调度策略 | 基于通信模式的差异化优先级分配+公平带宽共享+可选学习优化 | 任务通信模式标签+历史性能 | 每个任务的(priority, bandwidth_quota, path_set) |
| M5: QP抢占管理器 | 执行带宽调度 | 封装PeRF的IB_WR_WAIT/ENABLE/0_WAIT抢占原语 | 带宽配额变更指令 | 实际QP速率变化 |
| M6: 路径管理器 | 执行路径调度 | 基于Ethereal的确定性流分割+源路由，分配非冲突路径 | 路径分配指令 | QP-to-path映射 |

### 数据流

NCCL Trace拦截器 → 通信事件流 → 通信模式分类器 → 任务模式标签 → 资源分配器 → (优先级, 带宽配额, 路径) → QP抢占管理器/速率控制器/路径管理器 → RDMA QP实际操作

### 控制流

周期性（每training iteration或每100ms）：采集 → 分类 → 决策 → 执行。紧急拥塞事件触发异步快速路径（<1ms响应）

### 回退机制

调度决策引擎故障时回退到静态均分带宽（fair-share），保证基本可用性。

### 安全边界

调度器运行在特权守护进程中，训练任务无权修改调度策略；QP配额强制执行，恶意绕过受RDMA保护域限制。

## 9. 关键研究内容

### R1：多租户训练通信模式在线感知与表征

- **解决问题**：如何在不侵入训练代码的前提下，高效准确地在线感知多个训练任务的通信模式（迭代周期、burst/空闲交替、通信操作类型）？
- **方法**：LD_PRELOAD拦截NCCL API调用+RDMA计数器采集，设计低开销（<3% CPU）的轻量级通信模式表征框架
- **预期结果**：（1）通信模式感知精度>90%（以已知ground truth的合成负载验证）；（2）感知开销<3% CPU和<1%训练吞吐；（3）在3种以上的训练任务类型（大模型预训练、LoRA微调、CV训练）上验证表征的通用性

### R2：通信语义驱动的RDMA网络资源动态调度

- **解决问题**：如何将感知到的训练通信模式转化为有效的RDMA网络资源调度决策（差异化带宽分配、抢占式QP调度、冲突避免的路径分配）？
- **方法**：设计基于通信模式的差异化优先级分配算法+公平带宽共享策略；利用PeRF抢占原语（WAIT/ENABLE）实现QP级粒度调度；结合Ethereal的确定性路径分配避免ECMP冲突
- **预期结果**：（1）多任务并发时高优先级任务带宽退化<10%；（2）相比无隔离方案，最差任务JCT降低>30%；（3）相比静态均分带宽，集群网络利用率提升>15%

### R3：原型系统构建与实验验证

- **解决问题**：上述方法在真实RDMA环境中的可行性和有效性如何？与现有基线方案相比的优势有多大？
- **方法**：在8-64节点的RoCEv2集群上构建原型系统，使用真实分布式训练任务（GPT-2级别预训练+LoRA微调+ResNet-50训练）的多任务并发场景进行端到端验证
- **预期结果**：（1）开源原型系统代码；（2）在≥3种多任务并发场景下完成与基线方案（无隔离/静态均分/PeRF原始方案）的对比实验；（3）形成可复现的实验数据和benchmark套件

## 10. 验证计划

### Baseline
- 无隔离（原生RDMA共享）
- 静态均分带宽（Equal Share）
- PeRF原始方案（消息大小分类，不感知训练模式）

### 核心指标
- 任务级：JCT（作业完成时间）、训练吞吐（samples/sec）、MFU（Model FLOPS Utilization）
- 网络级：带宽退化率（%）、带宽公平性（Jain's Fairness Index）、尾延迟（P99 FCT）
- 系统级：调度开销（CPU%）、感知精度（%）、响应延迟（μs）

### 实验场景
- 场景1：1个大模型预训练 + 3个LoRA微调 并发（典型云智算场景）
- 场景2：2个同规模训练任务 并发（公平性压力测试）
- 场景3：1个高优先级 + 5个低优先级 并发（QoS差异化测试）

### 模块级验证
- M1+M2：通信模式感知精度（使用已知trace的合成负载）
- M3：分类准确率（burst/空闲/操作类型三分类）
- M4+M5：单QP抢占延迟和带宽稳定性

### 端到端验证
在16/32/64节点三种规模下，场景1-3的端到端训练吞吐和JCT对比

## 12. 风险与待确认问题

1. **项目类型确认**：当前按 NSFC 青年项目规划（3项研究内容，8-64节点实验）。若实际为面上项目，需扩展R2（增加智能调度方法深度）和R3（增加大规模仿真）
2. **实验环境**：方案假设有RoCEv2 RDMA集群可访问（8-64节点）。若无物理集群，R3需降级为仿真+小规模验证（4-8节点），需在方案中标注此风险
3. **NCCL拦截可行性**：LD_PRELOAD劫持NCCL API的技术可行性需在实际NCCL版本上验证——不同NCCL版本的内部API可能变化
4. **PeRF原语可用性**：PeRF的抢占原语（WAIT/ENABLE）需要RDMA支持相应verbs，需确认目标集群的RNIC和OFED驱动版本支持
5. **青年项目"智能"的表述**：NSFC对"智能"方法的期望程度——是否需要深度学习/强化学习方法，还是启发式/规则式调度即可满足"智能"的要求
